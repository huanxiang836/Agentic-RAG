### 样本 1

- 问题：请从职责边界、协作方式和适用场景三个角度，比较 A2A 和 Multi-agent 的差异。
- 参考答案：A2A 更强调跨系统、跨框架的标准通信与调用；Multi-agent 更强调同一应用内多个 Agent 的分工协作；前者偏对外互联，后者偏内部编排；实际选择取决于你是要连接外部 Agent，还是组织内部专家链路。
- 模型回答：基于知识库内容，A2A（Agent-to-Agent）与 Multi-agent（多智能体）在职责边界、协作方式和适用场景上存在显著差异。以下是具体的比较分析：

### 1. 职责边界
*   **Multi-agent**：聚焦于**单一应用或系统内部**的任务拆解。其职责边界基于“专业能力”进行划分（例如分为规划器、研究员、数学专家等）。这些 Agent 通常运行在同一个底层框架内，共享系统的上下文环境。
*   **A2A**：聚焦于**跨系统、跨组织**的独立智能体交互。其职责边界基于“物理部署或所属实体”划分（如不同的机器、不同的公司）。每个参与协作的 Agent 都是一个独立自治的实体，拥有自己的上下文和运行环境。

### 2. 协作方式
*   **Multi-agent**：依赖框架内部的编排机制进行协同，主要有两种模式且可混合使用：
    *   **工具调用（Agent Tool）**：采用集中控制的工作流，主 Agent 将子 Agent 作为工具来调用，以执行专门任务。
    *   **交接（Handoffs）**：进行 Agent 之间的控制权切换，支持 Agent 直接与用户交互，或在专家之间进行复杂的、类人的对话。
*   **A2A**：依赖**标准化的网络通信协议**（如 Google 推出的 Agent2Agent 协议）进行远程通信。它通过定义标准的注册、发现（如 AgentCard）和调用机制，解决异构智能体之间的通信问题，实现跨网络的无缝协作。

### 3. 适用场景
*   **Multi-agent**：适用于解决**单体应用内的复杂度瓶颈**。典型场景包括：
    *   单个 Agent 挂载的工具过多，导致难以做出正确的工具选择决策。
    *   任务的上下文或记忆增长过大，单个 Agent 难以有效跟踪。
    *   任务本身需要高度专业化的分工协作。
*   **A2A**：适用于解决**分布式环境下的互联互通问题**。典型场景包括：
    *   智能体应用需要进行分布式部署与远程通信。
    *   需要打通使用不同框架、不同技术栈构建的异构智能体。
    *   跨越不同公司或组织边界的智能体业务协作。 

总结来说，**Multi-agent 是一种应用架构设计模式**，用于在系统内部通过专业分工降低复杂度；而 **A2A 是一种跨系统的通信协议标准**，用于在分布式环境中连接彼此独立的智能体系统。两者并不冲突，在实际落地中，一个通过 A2A 协议对外提供服务的智能体，其内部可能正是由 Multi-agent 架构驱动的。

### 样本 2

- 问题：A2A 架构里 Server、Registry、Discovery 各自承担什么职责，三者如何配合？
- 参考答案：A2A Server 负责把本地 ReactAgent 暴露成可调用服务；Registry 负责登记和保存 AgentCard；Discovery 负责按需发现可用 Agent；三者组合起来形成注册、发现、调用的闭环。
- 模型回答：在 Spring AI Alibaba 的 A2A（Agent-to-Agent）架构中，Server、Registry 和 Discovery 是三个核心组件，它们各自承担不同的职责并紧密配合，以实现智能体之间的注册、发现与通信。

### 各自职责

1. **A2A Server（服务提供者）**
   * **职责**：负责将本地的智能体（如 `ReactAgent`）暴露为标准的 A2A 服务。
   * **作用**：它定义了本地 Agent 的能力、名称、描述等信息（即 AgentCard），并作为服务端接收和处理来自其他 Agent 或 Client 的请求。

2. **A2A Registry（注册中心）**
   * **职责**：作为 Agent 的全局注册中心（目前支持 Nacos 等）。
   * **作用**：提供 A2A AgentCard 模型的存储与推送支持，集中管理所有可用 Agent 的元数据信息。

3. **A2A Discovery（服务发现）**
   * **职责**：提供 Agent 的发现机制（作为服务消费者）。
   * **作用**：允许系统通过 `AgentCardProvider` 查询和发现网络中其他可用的 Agent，并支持在调用时实现负载均衡。

### 三者如何配合

这三者通过“注册-发现-调用”的机制协同工作，具体流程如下：

1. **服务注册**：
   当 A2A Server 启动时，如果开启了注册功能（`registry.enabled: true`），它会自动将本地 Agent 的元数据（AgentCard）注册到 A2A Registry 中。
2. **服务订阅与发现**：
   当 A2A Client（或另一个作为消费者的 Agent）需要调用外部能力时，如果开启了发现功能（`discovery.enabled: true`），它会通过 A2A Discovery 机制自动向 A2A Registry 订阅并拉取可用的 AgentCard 列表。
3. **负载均衡与调用**：
   A2A Client 获取到目标 Agent 的信息后，可以直接向对应的 A2A Server 发起调用。在此过程中，Discovery 机制还能帮助 Client 实现 AgentCard 调用的负载均衡，确保请求被合理分发。 

通过这种架构，不同的 Agent 可以动态地加入或退出网络，彼此之间无需硬编码对方的地址，即可实现灵活的互联互通与能力复用。

### 样本 4

- 问题：在配置 A2A Server 和 Nacos 时，哪些字段最容易配错，为什么？
- 参考答案：最容易配错的是 ReactAgent Bean 的 name、server.card.name 以及 Nacos 连接信息；因为这些字段直接决定 AgentCard 是否能正确生成、注册和被发现。
- 模型回答：在配置 A2A Server 和 Nacos 时，以下几个字段最容易配错：

1. **`registry.enabled`**
   * **为什么容易配错**：开发者在配置时可能会忽略显式开启注册功能。
   * **配错后果**：该字段必须配置为 `true` 才能将本地 Agent 注册到 Nacos（作为服务提供者）。如果未配置，会导致“Agent 没有注册到 Nacos”的故障，其他服务将无法发现该 Agent。

2. **`discovery.enabled`**
   * **为什么容易配错**：与注册功能类似，在需要调用其他 Agent 时容易被遗漏。
   * **配错后果**：必须配置为 `true` 才能启用服务发现（作为服务消费者）。如果未开启，A2A Client 将无法通过 AgentCardProvider 发现和订阅其他可用的 Agent，也就无法实现 AgentCard 调用的负载均衡。

3. **`server.card.name`**
   * **为什么容易配错**：这是一个需要与代码逻辑强关联的字段，容易在配置文件和代码定义之间产生不一致或拼写错误。
   * **配错后果**：该字段的值**必须**与代码中 `ReactAgent` Bean 的 `name` 保持完全一致。如果配错，会导致 Agent 身份标识无法正确匹配。

4. **基础连接配置（`server-addr`, `username`, `password`）**
   * **为什么容易配错**：涉及网络地址和鉴权信息，在不同环境（本地、测试、生产）切换时容易填错。
   * **配错后果**：如果地址或账号密码错误，应用将无法连接到 Nacos 服务器，导致 Nacos Registry 自动配置无法生效。在排查“Agent 没有注册到 Nacos”的问题时，通常需要优先验证这些基础连接配置是否正确。 

建议在配置时仔细检查注册与发现的开关是否按需打开，并确保 Agent 名称与代码中的 Bean 名称严格对应。

### 样本 5

- 问题：如果要用 AgentCardProvider 调用远程 Agent，整体链路应该怎么理解？
- 参考答案：先从注册中心获取目标 AgentCard，再拿到远程 Agent 的地址和能力描述，最后按 A2A 协议发起调用；本质上它把发现和调用拆成了两个明确步骤。
- 模型回答：使用 `AgentCardProvider` 调用远程 Agent 的整体链路可以划分为以下四个核心阶段：

### 1. 发现阶段 (Discovery)
首先，系统需要通过 `AgentCardProvider` 从注册中心（例如 Nacos）自动发现并获取目标远程 Agent 的元数据信息（即 `AgentCard`）。这一步使得调用方无需硬编码远程 Agent 的具体网络地址，实现了服务发现与解耦。

### 2. 构建阶段 (Build)
获取到发现能力后，使用 `A2aRemoteAgent` 的构建器（Builder）来实例化一个远程代理对象。在这个阶段，你需要指定该远程 Agent 的名称、描述，并将 `AgentCardProvider` 注入进去，以便它在后续调用时能动态解析端点。

### 3. 调用阶段 (Invocation)
通过构建好的 `A2aRemoteAgent` 实例，调用其 `invoke` 方法并传入具体的任务指令（Prompt）。在底层，框架会根据 `AgentCard` 中解析出的 REST API 端点，通过 A2A（Agent-to-Agent）协议向远程 Agent 发起网络请求。

### 4. 结果处理阶段 (Result Handling)
远程 Agent 处理完毕后，会返回一个包含执行状态和结果的对象（通常为 `Optional<OverAllState>`）。调用方可以通过解析这个状态对象，提取出最终的输出内容（如 `state.value("output")`）进行后续业务处理。

### 💡 链路代码示例

以下是该链路在 Spring AI Alibaba 中的典型代码实现：

```java
import com.alibaba.cloud.ai.graph.agent.a2a.A2aRemoteAgent;
import com.alibaba.cloud.ai.graph.agent.a2a.AgentCardProvider;
import com.alibaba.cloud.ai.graph.OverAllState;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Component;
import java.util.Optional;

@Component
public class A2AExample {

    private final AgentCardProvider agentCardProvider;

    @Autowired
    public A2AExample(AgentCardProvider agentCardProvider) {
        this.agentCardProvider = agentCardProvider;
    }

    public void callRemoteAgent() {
        // 【1 & 2. 发现与构建】通过 AgentCardProvider 从 Nacos 自动获取 AgentCard 并构建远程代理
        A2aRemoteAgent remote = A2aRemoteAgent.builder()
                .name("data_analysis_agent")
                .agentCardProvider(agentCardProvider)  
                .description("数据分析远程代理")
                .build();

        // 【3. 远程调用】发起 A2A 调用
        Optional<OverAllState> result = remote.invoke("请根据季度数据给出同比与环比分析概要。");

        // 【4. 结果处理】解析返回的状态数据
        result.ifPresent(state -> {
            System.out.println("调用成功: " + state.value("output"));
        });
    }
}
```

**排错提示**：如果在调用阶段失败，通常需要确认目标 Agent 的 REST API 端点是否可访问、检查网络连接和防火墙配置，并查看 A2A 消息传输日志以定位问题。

### 样本 6

- 问题：请比较 Tool Calling 和 Handoffs 两种多 Agent 模式，分别适合什么任务？
- 参考答案：Tool Calling 适合集中式编排，由主 Agent 统一决定何时调用子 Agent；Handoffs 适合把当前会话控制权直接交给更合适的专家 Agent；前者偏工作流，后者偏角色接管。
- 模型回答：Tool Calling（工具调用）和 Handoffs（交接）是多 Agent 系统中两种常见的协作模式，它们在控制流、交互方式和适用场景上有显著的区别。以下是两者的详细比较：

### 1. Tool Calling（工具调用）模式
* **核心机制**：采用“控制器-工具”架构。一个主 Agent（控制器）负责接收用户输入并管理整体编排，它将其他子 Agent 视为“工具”（AgentTool）。当需要执行特定任务时，控制器调用相应的子 Agent，子 Agent 执行完毕后仅将结果返回给控制器，由控制器决定下一步操作或生成最终回复。
* **交互特点**：作为工具使用的子 Agent **不直接与用户对话**，它们的角色纯粹是执行任务并返回结果。整个工作流由控制器集中管理。
* **适合的任务**：
  * **需要集中控制工作流的任务**：主 Agent 需要严格把控任务进度和最终输出质量。
  * **后台处理与任务拆解**：例如主 Agent 调用一个“数据分析 Agent”处理表格，再调用一个“翻译 Agent”翻译文本，最后由主 Agent 汇总结果给用户。
  * **无需多轮对话的子任务**：子任务目标明确，执行完即可返回，不需要向用户追问或解释。

### 2. Handoffs（交接）模式
* **核心机制**：采用“控制权传递”架构。Agent 之间可以直接将控制权和上下文状态传递给彼此。当当前 Agent 判断需要其他领域的专家协助时，会触发交接，将主导权移交给下一个 Agent。
* **交互特点**：系统中的“活动” Agent 会发生动态变化，**新接手的 Agent 会直接与用户进行交互**，直到它完成任务或决定再次交接。
* **适合的任务**：
  * **需要 Agent 直接与用户交互的场景**：例如智能客服系统，当“售前 Agent”遇到售后问题时，直接将对话控制权交接给“售后 Agent”，由售后 Agent 继续与用户沟通。
  * **专家之间复杂的、类人对话**：多个不同领域的专家 Agent 需要轮流与用户对话、收集信息或提供专业建议（如医疗问诊中的分诊与专科医生流转）。
  * **基于对话状态的流转**：工作流不是由单一节点强控，而是随着对话的深入在不同角色间自然流转。

### 3. 核心对比总结

| 比较维度 | Tool Calling (工具调用) | Handoffs (交接) |
| :--- | :--- | :--- |
| **工作流控制** | ✅ **集中控制**（由控制器 Agent 统筹） | ❌ **分散控制**（控制权在 Agent 间传递） |
| **用户交互** | ❌ **否**（子 Agent 仅返回结果给控制器） | ✅ **是**（当前活动 Agent 直接面对用户） |
| **对话复杂度** | ❌ **有限**（子 Agent 之间无直接对话） | ✅ **强**（支持专家间复杂的、类人对话流转） |

### 💡 混合使用建议
在实际的复杂业务中，这两种模式并不是互斥的，通常可以**混合使用**：
你可以使用 **Handoffs** 模式来处理前端与用户的交互流转（例如从“需求收集 Agent”交接给“方案生成 Agent”），同时让每个 Agent 在内部使用 **Tool Calling** 模式，将一些专门的、无需对话的底层任务（如查询数据库、调用外部 API）作为工具来调用执行。

### 样本 7

- 问题：从路由、编排和扩展性三个角度，Multi-agent 常见模式应该怎么选？
- 参考答案：如果任务需要明确编排和结果汇总，优先 Tool Calling；如果需要连续对话中切换专家，优先 Handoffs；如果任务分发要靠输入语义自动决定，优先 LlmRoutingAgent。
- 模型回答：在 Multi-agent 系统中，常见的模式主要有 **Tool Calling** 和 **Handoffs** 两种。从路由、编排和扩展性三个角度来看，它们的特征和选择建议如下：

### 1. 路由 (Routing) 角度
*   **Tool Calling（集中式路由）**：
    *   **机制**：由一个 Supervisor Agent（主管代理）负责所有的路由决策。它会根据用户的输入内容以及子 Agent 的描述，智能、自动地选择最合适的 Agent。
    *   **特点**：每次请求通常只路由到一个 Agent 执行，路由权高度集中。
*   **Handoffs（去中心化路由）**：
    *   **机制**：没有统一的调度中心，而是由当前的活跃 Agent 根据对话上下文，自主决定将控制权转移给另一个 Agent。
    *   **特点**：路由是动态交接的，Agent 之间可以互相改变当前由谁来担当活跃角色。

### 2. 编排 (Orchestration) 角度
*   **Tool Calling（任务编排）**：
    *   **机制**：Supervisor Agent 掌控全局工作流。子 Agent 仅仅作为“工具”被调用，它们只负责执行具体任务并将结果返回给 Supervisor，**不直接与用户对话**。
    *   **适用场景**：适合结构化工作流、明确的任务编排（如：主管分配数据查询任务给数据库 Agent，汇总后再回复用户）。
*   **Handoffs（专家接管）**：
    *   **机制**：当控制权发生转移时，活动 Agent 随之变更，新的 Agent 会**直接与用户进行交互**。
    *   **适用场景**：适合跨领域对话、需要专家直接接管的场景（如：从“售前咨询 Agent”无缝转接给“售后技术 Agent”直接解答用户问题）。

### 3. 扩展性 (Scalability) 角度
*   **Tool Calling（描述驱动，极易扩展）**：

1. **需要压缩长对话与生成摘要**

2. **需要过滤冗余或无关消息**

   * **瞬时消息更新**：有时只需要在单次调用时临时修改或过滤消息（如上述的拦截器操作），这不会影响整体持久存储的短期记忆，下次推理调用也不会感知到这次临时的过滤动作。
   * **持久消息更新**：对于需要永久改变对话历史的情况（例如将长对话永久替换为摘要），则需要使用生命周期钩子（Hook）等机制来永久更新状态中保存的数据。

- 问题：请比较短期内存、长期内存和持久化 checkpoint 的职责边界。
- 参考答案：短期内存负责会话内的连续对话，checkpoint 负责把图状态持久化以便恢复，长期内存负责跨会话保存用户画像；三者分别对应当前上下文、恢复能力和跨会话记忆。
- 模型回答：在智能体（Agent）和图（Graph）工作流的架构中，短期内存、长期内存和持久化 Checkpoint 扮演着不同但相互协作的角色。它们的职责边界可以从**作用域、业务目的和底层实现**三个维度来划分：

### 1. 短期内存 (Short-term Memory)
*   **核心职责**：负责**会话级（Session-level）** 的上下文跟踪，使智能体能够记住当前对话的历史记录，从而支持多轮连贯的交互。
*   **作用域**：绑定到特定的会话（通常通过 `threadId` 标识）。当会话结束或过期时，短期内存通常不再被主动调用。
*   **业务场景**：用户在一次对话中提到的前文信息（例如：“你好，我是 Bob” -> “根据我的名字给我写首诗”）。
*   **实现方式**：在框架中，短期内存通常依赖于状态图（StateGraph）的配置，通过注册 Checkpointer（如 `MemorySaver`）并配合 `SaverConfig` 来实现。

### 2. 长期内存 (Long-term Memory)
*   **核心职责**：负责**跨会话（Cross-session）** 的持久化信息存储，用于构建用户画像、保存全局偏好或积累业务知识，提供高度个性化和上下文感知的体验。
*   **作用域**：绑定到特定的用户或实体（通常通过 `userId` 或命名空间标识），独立于任何单一的 `threadId` 存在。
*   **业务场景**：用户的长期偏好设置（如：主题设为暗黑模式、语言偏好为中文、时区为 Asia/Shanghai）。无论用户开启多少个新的对话线程，这些偏好都能被加载和应用。
*   **实现方式**：通常通过专门的存储组件（如 `MemoryStore`）实现，采用命名空间（Namespace）和键值对（如 `StoreItem`）的结构进行数据的存取和管理。

### 3. 持久化 Checkpoint (Persistence Checkpoint)
*   **核心职责**：作为**底层技术机制**，负责在图（Graph）执行的各个节点保存状态快照（Snapshot）。它确保了工作流的可靠性、可恢复性以及状态的可追溯性。
*   **作用域**：作用于图执行的生命周期。
*   **业务场景**：
    *   **支撑短期内存**：通过保存每次对话后的图状态，使得下一次请求能够接续之前的上下文。
    *   **容错与恢复**：如果节点执行失败，可以从上一个 Checkpoint 恢复执行。
    *   **时间旅行（Time-travel）**：允许开发者或系统回滚到图执行的任意历史状态。
*   **实现方式**：通过实现 Checkpointer 接口（如内存级别的 `MemorySaver`，或生产环境中的数据库 Saver），并将其配置到图的编译配置（`CompileConfig`）中。

### Checkpointer 在 Graph 里到底做了什么？

1. **保存状态快照（Checkpoint）**
   在 Graph 执行的每一步，Checkpointer 会自动保存当前图状态的“检查点”。每条 Checkpoint 中主要记录了两个关键信息：
   * **state**：当前步骤的完整状态值。
   * **nextNodeId**：图中接下来要执行的节点标识符。

2. **支持人机协作工作流（Human-in-the-loop）**
   因为 Checkpointer 记录了精确的状态和下一步节点，它允许 Graph 在执行过程中**中断**。人类可以在任何时间点查看当前状态，进行人工干预（如修改状态、批准操作），然后 Graph 能够基于更新后的状态**恢复执行**。

3. **跨交互的“记忆”保持（会话管理）**
   Checkpointer 允许在多次独立的交互之间保持记忆。通过为每次运行指定一个唯一的 `threadId`（会话 ID），Checkpointer 会将一系列检查点关联起来。在诸如多轮对话的场景中，后续消息发送到同一个 `threadId` 时，Graph 会自动加载之前的状态，从而拥有上下文记忆。

4. **历史追踪与恢复**
   Checkpointer 会保存状态的历史版本。这意味着你可以查看 Graph 运行过程中的历史轨迹，甚至在需要时回滚或恢复到某个特定的历史状态。

### Checkpointer 和普通状态变量有什么区别？

虽然它们都用于“存储数据”，但在 Graph 架构中，两者的定位和能力有本质区别：

| 维度 | 普通状态变量 (State) | Checkpointer (检查点机制) |
| :--- | :--- | :--- |
| **生命周期与存储** | 通常只存在于单次执行的内存中，Graph 运行结束即销毁。 | 提供**持久化存储**（如内存 `MemorySaver`、Redis 等），状态可跨越多次独立的 Graph 调用存活。 |

- 模型回答：从组件角度看，构建一个可用的知识库（通常基于 ETL 管道：Extract、Transform、Load）至少需要以下核心模块：

1. **数据加载模块（Extract / Reader）**
   - **作用**：负责连接不同的数据源（如本地文件系统、数据库、网页等），读取并提取原始文档数据。
   - **示例组件**：各类文档读取器（如 `TextReader`）。

2. **数据处理与分割模块（Transform / Splitter）**
   - **作用**：对加载的原始文档进行清洗、格式转换，并将其切分成大小合适的文本块（Chunks）。这一步是为了确保数据长度适合后续的模型处理，并提高检索的精准度。
   - **示例组件**：文本分割器（如 `TokenTextSplitter`）。

3. **向量化模块（Embedding）**
   - **作用**：将分割好的文本块转换为高维向量表示。这是让计算机能够理解和计算文本语义相似度的关键步骤。
   - **示例组件**：嵌入模型（Embedding Model）。

4. **存储与检索模块（Load / VectorStore）**
   - **作用**：负责持久化存储文本块、元数据及其对应的向量，并提供高效的相似度搜索（Similarity Search）接口。当用户发起查询时，该模块负责快速召回最相关的知识片段。
   - **示例组件**：向量数据库（如 `VectorStore`）。

### 样本 24

- 问题：两步 RAG、Agentic RAG 和混合 RAG 的区别是什么，怎么选？
- 参考答案：两步 RAG 是先检索后生成，简单且可预测；Agentic RAG 让 Agent 在推理过程中决定何时检索，更灵活；混合 RAG 在中间加入验证和后处理，适合对质量要求更高的场景。
- 模型回答：根据知识库内容，两步 RAG、Agentic RAG 和混合 RAG 在架构设计、执行流程和控制粒度上存在显著区别。以下是它们的详细对比及选择建议：

### 1. 核心区别

**两步 RAG (Two-Step RAG)**
*   **执行流程**：采用固定的管道设计，**检索步骤总是在生成步骤之前执行**。
*   **特点**：架构简单、执行过程高度可预测。通常通过开箱即用的组件（如 Spring AI 中的 Advisor）自动处理检索和上下文增强。
*   **控制粒度**：线性控制，缺乏中间反馈机制。

**Agentic RAG (智能体 RAG)**
*   **执行流程**：由 Agent（智能体）主导，没有固定的线性管道。
*   **特点**：具有极高的**灵活性**。Agent 可以根据当前状态自主决策，例如自主选择使用文档搜索、网络搜索等多种工具，并支持在推理循环中多次调用工具。
*   **控制粒度**：高度自治，但相对缺乏对单次生成质量的强制干预。

**混合 RAG (Hybrid RAG)**
*   **执行流程**：结合了两步 RAG 和 Agentic RAG 的特点，在 Agent 的自主执行过程中引入了**中间步骤**（如查询预处理、检索验证和生成后检查），并支持步骤间的多次迭代。
*   **特点**：兼具 Agentic RAG 的灵活性和两步 RAG 的质量控制。典型组件包括：
    *   **查询增强**：修改输入问题（如重写、生成变体或扩展上下文）以提高检索质量。通常在 Agent 开始时执行一次，以避免每次推理循环都调用，从而降低成本。
    *   **检索验证**：评估检索到的文档是否相关且充分，若不足则优化查询并再次检索。
    *   **答案验证**：检查生成答案的准确性、完整性及与源内容的一致性，必要时重新生成或修订。

---

### 2. 怎么选？

选择哪种架构主要取决于你的**业务场景复杂度**、**对答案质量的要求**以及**成本考量**：

*   **选择 两步 RAG，如果：**
    *   你的应用场景相对简单，用户的查询意图明确。
    *   检索相关文档是生成答案的明确且唯一的前提。
    *   你希望系统架构简单、易于维护，且执行结果高度可预测。
    *   *示例：企业内部规章制度问答、产品说明书查询。*

*   **选择 Agentic RAG，如果：**
    *   你需要处理复杂的任务，且需要跨多个异构信息源（如本地文档库 + 实时互联网搜索）获取信息。
    *   系统需要具备自主规划能力，能够根据初步检索结果决定下一步行动（如继续搜索或停止搜索）。
    *   *示例：需要综合多方信息进行市场调研分析的智能助手。*

*   **选择 混合 RAG，如果：**
    *   你既需要多工具检索的灵活性，又对最终答案的质量有严格要求。
    *   应用经常面临**模糊或不明确的查询**，需要系统先进行查询增强。
    *   系统属于**领域特定的问答系统**，要求高准确性，必须通过“检索验证”和“答案验证”等质量控制步骤来防止幻觉或信息遗漏。
    *   *示例：医疗、法律等容错率极低的专业领域问答系统，或需要处理复杂长尾问题的智能客服。*

### 样本 25

- 问题：Graph 引擎在 Agent 编排里扮演什么角色，为什么不是简单的顺序调用？
- 参考答案：Graph 引擎把 Agent 工作流抽象成节点和边，并通过状态驱动执行；它比顺序调用更适合条件分支、并行执行、回路和恢复。
- 模型回答：在 Agent 编排中，Graph 引擎（如 StateGraph）扮演着**核心调度与状态管理中心**的角色。它不仅允许将单个 Agent（如 `ReactAgent`）作为节点（Node）集成以发挥其推理和工具调用能力，还为多智能体系统提供了强大的组合与协同能力。

之所以不采用简单的顺序调用，是因为真实的业务场景往往具有高度的复杂性和动态性，Graph 引擎能够解决顺序调用无法处理的以下问题：

1. **动态路由与条件决策**
   简单的顺序调用只能按照固定的线性步骤执行。而 Graph 引擎支持**条件边（Conditional Edges）**，允许系统根据当前的运行状态或中间结果动态决定下一步走向。例如在 Supervisor（监督者）模式中，Supervisor 节点可以评估任务需求，动态将任务路由给不同的 Worker（如研究员或程序员），Worker 完成后还能将控制权交还给 Supervisor 进行下一步决策，直到任务最终完成。
2. **支持复杂的协作模式**
   除了基础的顺序代理（SequentialAgent），Graph 引擎原生支持或易于构建多种高级工作流模式，包括：
   * **并行代理（ParallelAgent）**：多个 Agent 同时处理不同子任务，提升效率。
   * **路由代理（RoutingAgent）**：根据意图将请求分发给最合适的专业 Agent。
   * **循环代理（LoopAgent）**：支持多步推理和反复迭代，适用于需要不断反思和修正的复杂任务。
3. **更直接的状态控制与高可靠性**
   在需要超高可靠性、大量自定义逻辑或需要精确控制延迟的场景中，直接使用 Graph API 可以提供更细粒度的状态管理。开发者可以清晰地定义节点之间的数据流转和状态更新策略，确保复杂工作流的稳定运行。

总结来说，简单的顺序调用适用于单一、线性的任务，而 Graph 引擎通过图结构（节点与边）打破了线性限制，赋予了 Agent 系统**动态决策、循环迭代、并行处理**的能力，从而能够应对需要多步骤推理和复杂协作的真实场景。
